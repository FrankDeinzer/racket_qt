## 3. Checkpoint D-0 — Layout-Refactor

### Problem

Nach Checkpoint C ordnete der Shim Canvas und Button über ein `QVBoxLayout` an. Rackets eigenes Layout-System (`make-top-container%` / `make-item%`) berechnet Positionen und ruft `set-size` auf — diese wurden ignoriert. Panels (`panel%`) hatten kein echtes Qt-Widget.

### Lösung

**Shim (`qt-shim/src/shim.cpp`):**
- `#include <QVBoxLayout>` entfernt; kein `QVBoxLayout` mehr in `RacketWindow`
- Neue Funktion `shim_window_get_content_widget(win*)` → `QWidget*` (central widget von `QMainWindow`)
- Neue Funktion `shim_widget_set_geometry(widget*, x, y, w, h)` → `widget->setGeometry(...)`
- Neue Funktion `shim_panel_create(parent*)` → erzeugt `QWidget` mit gegebenem Parent
- `shim_canvas_create` und `shim_button_create` nehmen jetzt `QWidget*` statt `RacketWindow*` als Parent

**Racket (`wx/qt/`):**
- `frame.rkt`: speichert `content-handle` via `shim_window_get_content_widget`; `get-content-hwnd` gibt diesen zurück
- `window.rkt`: Default-`get-content-hwnd` gibt `handle` zurück
- `canvas.rkt`, `button.rkt`: Parent-Handle via `(send parent get-content-hwnd)`; `set-size`-Override ruft `shim_widget_set_geometry` auf
- `panel.rkt`: komplett neu — erzeugt echten `QWidget` via `shim_panel_create`; `get-content-hwnd` gibt eigenes Handle zurück, damit Kinder korrekt im Qt-Parent-Baum landen

**Ergebnis:** `examples/hello.rkt` funktioniert nach wie vor; Canvas und Button werden jetzt von Rackets Layout-Engine positioniert, nicht von Qt.

## 4. Checkpoint D-1 — Eingabe-Rückgrat

### Neue Callback-Typen im Shim

```cpp
typedef void (*shim_mouse_cb_t)(void* ud, int type, int x, int y, int buttons, int mods);
typedef void (*shim_key_cb_t)(void* ud, int type, int key, int text_char, int mods);
typedef void (*shim_focus_cb_t)(void* ud, int gained);
```

`RacketCanvas` hat jetzt `setFocusPolicy(Qt::StrongFocus)` und `setMouseTracking(true)`. Neue Shim-Exporte: `shim_canvas_set_mouse_cb`, `shim_canvas_set_key_cb`, `shim_canvas_set_focus_cb`, `shim_widget_set_focus`.

**Wichtige Qt-6-Besonderheit:** `e->button()` (Singular) — der auslösende Button bei Press/Release; `e->buttons()` (Plural) — alle gehaltenen Buttons bei Move. Release-Events in Qt löschen den Button aus `buttons()` bereits *vor* dem Event, daher muss `e->button()` für Press/Release verwendet werden.

### `key-map.rkt` (neues File)

Qt::Key → Racket key-code Mapping:
- Sonderzeichen-Tabelle (Escape, Tab, Backspace, Return, Pfeile, F1–F24, Modifier-Keys, …)
- Keys ≥ `0x01000000` sind Sondertasten; darunter printable chars via `text-char`-Unicode
- Hilfsfunktionen: `qt-mods->shift?`, `qt-mods->control?`, `qt-mods->alt?`, `qt-mods->meta?`
- Button-Bitmask-Helfer: `qt-buttons->left?`, `qt-buttons->middle?`, `qt-buttons->right?`

### `window.rkt` — Dispatch-Protokoll

Hinzugefügt:
```racket
(define/public (dispatch-on-char e just-pre?)
  (cond [(other-modal? this) #t]
        [(call-pre-on-char this e) #t]
        [just-pre? #f]
        [else (when (is-enabled-to-root?) (on-char e)) #t]))
(define/public (call-pre-on-char w e)
  (if (and parent (object? parent))
      (or (send parent call-pre-on-char w e) (pre-on-char w e))
      (pre-on-char w e)))
```
Analog `dispatch-on-event`. Spiegelt das Muster von win32/gtk/cocoa.

### `canvas.rkt` — Callbacks

Im Initializer von `base-canvas%`:
- `mouse-cb`: konvertiert Qt-type/buttons/mods → `new mouse-event%` mit Keyword-Args; postet via `queue-event`
- `key-cb`: konvertiert Qt::Key + text-char → `new key-event%` mit Keyword-Args; für Release: `set-key-release-code`
- `focus-cb`: dispatcht `on-set-focus`/`on-kill-focus`
- `set-focus` override: ruft `shim_widget_set_focus` auf

**Release-Button-Logik:** Beim Release gibt `buttons` nur den *auslösenden* Button (Bit gesetzt). Da der Button nach Release *nicht* mehr gedrückt ist, wird das Bit für `left-down?` etc. invertiert:
```racket
(define left-down? (if (= type 1) (not left?) left?))
```

### `examples/input.rkt` (neues Beispiel)

Subklasse von `canvas%` mit `define/override` für `on-event`, `on-char`, `on-focus`:
- Zeigt live: Mauskoordinaten + Event-Typ, letzte Taste + Modifikatoren, Fokus-Zustand
- Animierter Box via `(new timer% [interval 50] ...)` der ±3 px pro Tick wandert

### `tests/smoke.rkt` (neue Tests)

3 `rackunit`-Testfälle:
1. `frame%` + `canvas%` + `button%` erzeugen und zeigen
2. Timer feuert 5× in < 1 Sekunde (`pump-n` mit `(sleep 0.01)` + `(yield)` — beide nötig)
3. Subklasse von `canvas%` mit `on-char`-Override empfängt Key-Events

**Timer-Lesson:** `(sleep 0.01)` allein yield den OS-Thread, verarbeitet aber keine Eventspace-Events. `(yield)` nach dem Sleep nötig, damit der Handler-Thread die Event-Queue drainiert.

**Testergebnis:** 3/3 grün (`raco test tests/smoke.rkt`)

## 7. Erkenntnisse

### 7.1 `get-content-hwnd`-Protokoll

Für ein korrektes Qt-Parent-Kind-Verhältnis muss das Protokoll streng eingehalten werden:
- `frame%` → gibt `content-handle` zurück (das central `QWidget*` von `QMainWindow`)
- `panel%` → gibt eigenes `QWidget*` zurück
- alle anderen → Default in `window%` gibt `handle` zurück

Ohne dieses Protokoll würden Kinder (Canvas, Button) als direkte Kinder des `QMainWindow`-Fensters angelegt, was Qt nicht erwartet.

### 7.2 Zwei-Repo-Commit-Disziplin

Alle Änderungen in `wx/qt/` und `qt-shim/` landen in *zwei* Commits:
1. gui-Submodul (`FrankDeinzer/racket_gui`, Branch `qt-backend`)
2. Umbrella-Repo (`C:\src\racket_qt`, Branch `main`) — updated Submodul-Pointer

Commit in falscher Reihenfolge würde Umbrella auf alten Submodul-Stand zeigen lassen.

### 7.3 `(yield)` in Handler-Thread

Rackets Eventspace-Events werden über die Handler-Thread-Queue verarbeitet. Ein Thread im Handler-Modus muss `(yield)` aufrufen, um die Queue zu drainieren — `(sleep ...)` allein tut das nicht. Das gilt für Timer-Events und alle anderen `queue-event`-geposteten Events.

### 7.4 Was schwieriger war als erwartet

- **Mouse-Release-Buttons in Qt:** `e->buttons()` vs. `e->button()` ist eine Qt-API-Falle, die in der Dokumentation nur implizit erwähnt wird. Beim Release-Event ist der Button bereits aus `buttons()` verschwunden — man muss `button()` (Singular) nehmen.
- **Event-Konstruktoren mit Keyword-Args:** Die interleaved `(init [time-stamp 0])`-Parameter in `mouse-event%` machen die positionale Reihenfolge gefährlich. Keyword-Args sind deutlich robuster.
- **Timer im Test:** Das `(yield)`-Erfordernis in Racket-Handler-Threads ist nicht sofort offensichtlich — erforderte Debugging des Smoke-Tests.
